# ADR-008: Screen Layout for v1 (poc-purchasing-approval)

## Status

Accepted

## Related Diagrams

- Screen Flow (v1): [../diagrams/screen-layouts/screen-flow-v1.mmd](../diagrams/screen-layouts/screen-flow-v1.mmd)

## Context

poc-purchasing-approval v1 では、購買申請〜承認/差し戻しまでの最小フローを実装対象とする。

前提条件は以下の通り。

- ユーザー種別として「申請者」「承認者」が存在する
- 申請の作成は、申請者/承認者の双方が行える
- 承認者のみ、申請中の申請に対して「承認」「差し戻し」が可能
- 「申請詳細（申請内容確認）画面」を承認者が閲覧した場合のみ、AI要約が表示される
- 申請詳細画面では、申請者/承認者ともにコメントを残せる

また、v1では過剰な画面分割や機能の先行実装を避け、開発コストを抑えつつ拡張余地を残す。

## Decision

v1 の画面構成は **5画面** とし、承認アクション・AI要約・コメント機能は **申請詳細画面に集約**する。

### 採用する画面（v1）

1. Login画面
   - 認証のみ（ロールはログイン後にサーバ側で解決）

2. Home / Dashboard画面（申請一覧）
   - 申請一覧（table）を表示し、申請詳細へ遷移可能
   - 拡張性として、検索/フィルタ領域を追加しやすいレイアウトを前提にする（v1では最小実装）

3. 新規申請作成画面
   - 申請入力フォーム（申請者/承認者ともに利用可能）

4. 新規申請確認画面
   - 作成画面の入力後、「確認」ボタンで遷移し、内容を確認して確定

5. 申請詳細画面
   - 共通：申請内容・ステータス・コメント一覧/投稿
   - 承認者のみ：AI要約の表示、承認/差し戻しボタンの表示・実行

## Alternatives Considered

1. 「承認待ち一覧」「自分の申請一覧」などを別画面として分割する
   - 不採用理由：v1の画面数増加により実装/導線/認可制御が増え、コストが上がる。  
     一覧の用途分割は Home 画面内のタブ/フィルタで拡張可能。

2. 「コメント」専用画面を追加する
   - 不採用理由：詳細画面に統合できるため、画面追加のメリットが薄い。

3. 「AI要約」専用画面（生成履歴/再生成など）を追加する
   - 不採用理由：v1の要件は「承認者が詳細を見た場合のみ表示」であり、詳細画面内で十分。  
     追加画面はv2候補。

4. 新規申請確認画面を省略し、作成画面から直接作成する
   - 不採用理由：誤入力防止・確認フローの明確化のため、確認画面を残す。  
     （POCでも「入力→確認→作成」の体験を優先）

## Consequences

### Positive

- v1の必要要件（申請作成・承認/差し戻し・AI要約・コメント）を満たしつつ、画面数を最小化できる
- 承認アクション、AI要約、コメントを「申請詳細」に集約することで導線と権限制御が単純になる
- Home を入口に集約することで、将来的な検索/フィルタ/タブ追加を画面増加なしで行える

### Negative

- Home 画面が一覧用途の中心となり、将来の機能追加でUIが肥大化しやすい（タブ/フィルタ設計が重要）
- v1では検索/担当購買種別などの補助情報が簡素になり、運用上の利便性は限定的
- AI要約の生成・履歴・再生成などを扱う拡張は v1 では扱わず、要件追加時に設計見直しが必要

## Notes (Optional)

- Home 画面の拡張は「画面追加」ではなく、基本はタブ/フィルタで吸収する方針とする
- v2候補：
  - 検索（キーワード/ステータス/購買種別/期間など）
  - 承認者の担当購買種別の可視化（表示強化）
  - 申請の編集/取消
  - AI要約の履歴管理・再生成・評価（フィードバック）など
